# DECISIONS.md

## Current Repository Decisions

- Architecture style: layered DDD backend structure with explicit `presentation -> application -> domain <- infrastructure` boundaries.
- Database: PostgreSQL only.
- ORM: Prisma.
- Linting and formatting: Biome.
- Backend framework: NestJS is preserved as the presentation/composition shell for the DDD backend.
- Backend contract preservation: the refactor keeps the existing `/api` route family, error envelope, RBAC semantics, JWT access token flow, and refresh-cookie transport.
- Reports route family: analytics stay exposed under `/api/reports/*`, not `/api/analytics/*`.
- Backend DDD placement:
  - `src/presentation`: Nest modules, controllers, HTTP DTOs
  - `src/application`: use cases, orchestration, presenter-facing DTO mapping
  - `src/domain`: aggregates, value objects, policies, repository/algorithm ports
  - `src/infrastructure`: Prisma repositories, JWT, bcrypt, optimization adapters
- Shared backend foundation: `src/common` centralizes DTOs, guards, decorators, filters, pipes, validators, and cross-cutting errors.
- API validation strategy: global `ValidationPipe` uses `whitelist`, `forbidNonWhitelisted`, and `transform`, with errors normalized to the required `error.code`, `error.message`, `error.details` shape.
- UUID parameter policy: every public `:id` route parameter is validated as UUID at the controller boundary.
- Date-range policy:
  - `startDate > endDate` is rejected as a validation error before any repository query runs
  - date-only filters use Sao Paulo local day boundaries instead of UTC-midnight boundaries
- Money policy: order and pricing calculations use integer cents inside domain/application logic and convert only at persistence/API boundaries.
- Auth contract: Bearer access token plus refresh-token rotation via `httpOnly` cookie.
- Access token lifetime: `15m`.
- Refresh token lifetime: `7d`.
- Refresh token persistence strategy: refresh sessions are stored server-side in PostgreSQL and keyed by `jti`.
- Refresh rotation hardening:
  - login normalizes email with trim + lowercase before lookup
  - refresh rotation is transactional: the previous session is revoked and the replacement session is created in one database transaction
  - refresh-token reuse stays mapped to `INVALID_REFRESH_TOKEN`
  - expired/revoked refresh sessions are opportunistically cleaned during login and refresh activity
- Refresh-token cookie scope: the cookie path is `/api/auth`, not `/api/auth/refresh`, so the same cookie can be used for both refresh rotation and server-side logout revocation.
- Roles: `admin` has operational access; `viewer` is read-only for products, orders, and reports only.
- Rate limiting strategy: in-memory per-IP guard implementing `100 requests per minute per IP`. This is sufficient for the take-home and local Docker flow, but not meant as a multi-instance production strategy.
- Error handling strategy: business-specific errors use `AppException`; framework and Prisma errors are normalized by a global exception filter.
- Orders audit model: `deliveredAt` and `order_status_events` are part of the reference model for delivery-time analytics and auditability.
- Order detail contract: `GET /api/orders/:id` stays aligned with the list/detail operational payload and does not expose persisted status history publicly.
- Order consistency hardening:
  - duplicate `productId` entries are rejected during creation
  - missing or unavailable products stay mapped to `UNAVAILABLE_PRODUCT`
  - order creation validates products and persists the order in one transaction
  - courier availability checks and assignment writes run in one transaction
  - targeted optimistic concurrency is used on status-changing and assignment writes to reject stale concurrent updates with conflict semantics
  - the `deliveredAt` timestamp and the `delivered` status event reuse the exact same timestamp value
- Product write contract:
  - `PATCH /api/products/:id` is the primary partial-edit contract
  - `PUT /api/products/:id` remains as a compatibility alias during the refactor
  - `PATCH /api/products/:id/availability` is the dedicated availability toggle contract
- Query conventions:
  - products: reusable pagination DTO plus `search`, `category`, `isAvailable`, `sortBy`, `sortOrder`
  - orders: reusable pagination DTO plus `status`, `startDate`, `endDate`, `sortBy`, `sortOrder`
  - reports: reusable date-range DTO for analytics endpoints
- Optimization algorithm: `POST /api/orders/optimize-assignment` uses the Hungarian algorithm over a padded square cost matrix built from Haversine distances.
- Optimization eligibility:
  - input includes only `ready` orders and couriers that are active, not already tied to a `delivering` order, and have both current coordinates filled
  - couriers without coordinates are excluded silently from suggestions
  - assignments above `10 km` are rejected by a hard cap rather than merely ranked lower
- Optimization determinism:
  - lower distance always wins first
  - ties are broken by older ready orders first
  - courier ordering is stabilized for deterministic outputs
- Optimization acceptance flow: the client applies a suggestion with `PATCH /api/orders/:id/assign` followed by `PATCH /api/orders/:id/status` with `status=delivering`.
- Reports date semantics:
  - revenue, top-products, and average-delivery-time are filtered by `deliveredAt` and only consider delivered orders
  - orders-by-status is filtered by `createdAt`
  - average delivery time uses `createdAt -> deliveredAt`
  - delivered orders without an assigned courier are excluded from delivery-time aggregates overall and by vehicle type
  - report payload `startDate` and `endDate` reflect the exact effective window used in the query response
- Reports implementation strategy: aggregation is pushed down into Prisma `groupBy`, aggregate queries, and focused raw SQL instead of being assembled in memory in the application layer.
- Frontend architecture: React Router app shell split into public and protected layouts, with feature-specific pages instead of a monolithic `App.tsx`.
- Frontend data strategy:
  - auth session state remains in the existing auth context
  - domain data uses TanStack Query for fetching, mutation invalidation, caching, and background refresh
  - shared query-string building lives in one reusable service utility
- Frontend operational UX:
  - forms surface inline validation before submit
  - backend `error.details` are rendered more granularly instead of collapsed into a single generic message
  - `/orders` uses 30-second polling while the page is visible
  - optimization summary cards are recomputed after accepting/removing suggestions
  - non-admin users are redirected away from `/delivery-persons`
  - the top-bar date is computed during render instead of being frozen at module load
- Frontend testing stack: Vitest + Testing Library.
- Backend testing strategy: Jest unit tests plus Jest e2e tests for HTTP contracts, with concurrency/race-sensitive flows covered in the backend suite.
- Health check strategy: `/api/health` verifies Prisma reachability with a lightweight query instead of returning a static payload.
- Local environment strategy: the repository keeps the fast host-based workspace workflow and also ships a full `docker compose up --build` path for `postgres + backend-init + backend + frontend`.
- Prisma migration policy:
  - schema delivery uses committed Prisma migrations
  - `prisma db push` is not the supported delivery path anymore
  - container startup applies `prisma migrate deploy` before seeding and starting the API
- CI strategy: the repository ships a GitHub Actions workflow that installs dependencies, prepares environment variables, validates migrations, runs lint/typecheck/build, and executes frontend plus backend test suites.
- Linting tradeoff: Biome remains the single formatter/linter, but NestJS controller files are excluded from `biome check` because parameter decorators are unsupported and `unsafeParameterDecoratorsEnabled` is explicitly forbidden in this repository.
- Line endings policy: `.gitattributes` enforces LF for source and docs, while keeping Windows-native endings only for shell scripts that need them.
